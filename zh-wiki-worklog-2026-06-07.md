# 中文 Wiki 功能工作过程总结

日期：2026-06-07

## 背景

最初的问题是：GitNexus 当前生成的 wiki 主要是英文，希望增加“可选生成中文 wiki”的能力。

用户已有一份分析文档：

```text
/storage/emulated/0/TermuxHome/zh-wiki-plan.md
```

该文档提出了“方案 B + C”：让页面标题、模块名、正文和 Mermaid 标签全部中文化，并考虑修改 `slugify()` 支持中文文件名。

## 初始分析

首先读取了用户提供的 `zh-wiki-plan.md`，随后对照本地 GitNexus 源码确认实际实现。

确认到的事实：

- 实际源码路径是 `gitnexus/gitnexus/src/...`，方案文档中的 `src/...` 少了一层目录。
- `WikiGenerator` 已有 `--lang` 参数。
- `buildSystemPrompt()` 当前会要求正文使用指定语言，但明确说明 H1 标题仍保持英文。
- `generateLeafPage()`、`generateParentPage()`、`generateOverview()` 都会在代码里硬编码插入英文 H1。
- grouping 阶段没有使用 `buildSystemPrompt()`，模块名默认更容易保持英文。
- `slugify()` 当前只保留 ASCII 字母和数字。
- HTML viewer 里还有 `Overview`、`Modules`、`Generated by GitNexus`、`Page not found` 等硬编码英文 UI。

## 对原方案的判断

原方案的大方向是正确的：确实需要处理标题、模块名、正文、Mermaid 标签和 viewer 文案。

但原方案存在几个风险：

- 不应把 `prompts.ts` 全部改成中文，否则会影响默认英文 wiki 和其他语言 wiki。
- 不建议默认让中文模块名直接进入 slug 和文件名，因为会影响跨平台路径、URL hash、GitHub/Gist 浏览、增量更新和链接稳定性。
- grouping 阶段直接让 JSON key 中文化可行但风险较高，因为当前模块名、slug、meta 和增量更新之间耦合较紧。
- H1 提取逻辑必须只接受首行 H1，否则可能误删正文中间的标题。
- HTML viewer 的英文 UI 是原方案漏项。

因此最终采用了更稳的设计：中文显示名和稳定 slug 分离。

## 最终设计

目标是：默认行为不变，仅在用户显式指定中文语言时启用中文 wiki。

中文模式支持这些入口：

```bash
gitnexus wiki --lang chinese
gitnexus wiki --lang zh
gitnexus wiki --lang zh-CN
gitnexus wiki --lang 中文
gitnexus wiki --lang 简体中文
```

核心设计：

- grouping 仍按原逻辑运行，保持文件分组和 slug 稳定。
- 中文模式下，分组完成后额外调用一次 LLM，把模块显示名翻译成简体中文。
- `node.name` 用作可见显示名，可以是中文。
- `node.slug` 继续保持 ASCII，用作 `.md` 文件名、Markdown 链接目标和 HTML hash。
- 页面正文、标题、Mermaid 标签使用中文。
- 函数名、类名、路径、命令、公式、配置 key、链接目标保持原样。

示例输出目标：

```md
# 认证模块

该模块负责用户登录和会话管理，核心入口包括 `loginUser()` 和 `createSession()`。

相关源码位于 `src/auth/session.ts`。
```

文件名仍保持：

```text
auth.md
```

## 实现过程

### 1. 语言归一

在 `generator.ts` 中新增 `CHINESE_LANG = 'zh-CN'`。

`effectiveLang()` 现在会把以下输入统一为 `zh-CN`：

- `chinese`
- `zh`
- `zh-CN`
- `zh_cn`
- `中文`
- `简体中文`

这样同一个中文 wiki 不会因为用户这次输入 `中文`、下次输入 `zh-CN` 而被当成不同语言。

### 2. Prompt 语言约束

修改 `buildSystemPrompt()`，删除原来“H1 保持英文”的说明。

新的语言约束要求：

- 文档正文、小节标题、页面标题、Mermaid 标签使用目标语言。
- 保留代码标识符、函数名、类名、文件路径、包名、CLI 命令、import path、配置 key、公式、方程和 Markdown 链接目标。
- 不翻译真实 API 名称和源码。

### 3. 模块显示名本地化

新增 `localizeModuleTreeNames()`。

流程：

1. 只有 `effectiveLang() === 'zh-CN'` 时执行。
2. 收集所有模块节点，包括父模块和子模块。
3. 给 LLM 输入 `slug`、当前模块名和部分文件路径。
4. 要求 LLM 返回严格 JSON：`{ "auth": "认证模块" }`。
5. 只更新 `node.name`。
6. 不更新 `node.slug`。
7. 如果本地化失败，保留原英文模块名，生成流程继续。

这个设计保证中文显示名是可选增强，不会破坏原来的分组和链接稳定性。

### 4. 页面 H1 处理

新增 `composePageContent()`。

行为：

- 如果 LLM 输出首行是 `# 标题`，使用该标题。
- 如果 LLM 没有输出 H1，则使用 fallback 标题。
- 删除重复 H1。
- 最后仍经过 `sanitizeMermaidMarkdown()`。

这解决了原来代码强制插入英文 H1 的问题。

### 5. 父页和 overview 链接稳定性

修改 `PARENT_USER_PROMPT` 和 `OVERVIEW_USER_PROMPT`。

新增真实链接列表：

```md
- 认证模块: [认证模块](auth.md)
```

这样即使页面显示名是中文，LLM 也能使用正确的 ASCII slug 链接。

### 6. 增量更新兼容

中文显示名会进入 `moduleFiles` 和 `moduleTree` 的 `name` 字段。

因此增量更新不能再只依赖 `slugify(moduleName)` 查找节点。

新增 `findNodeByNameOrSlug()`：

- 先按 `node.name` 查找。
- 再按 `node.slug` 查找。
- 最后兼容旧逻辑，用 `slugify(nameOrSlug)` 查找。

### 7. HTML viewer 中文化

修改 `html-viewer.ts`。

新增 viewer label 配置，根据 `meta.lang` 判断：

- `zh-CN` -> 中文 UI
- 其他 -> 原英文 UI

中文模式下：

- `<html lang="zh-CN">`
- `Overview` -> `概览`
- `Modules` -> `模块`
- `Generated by GitNexus` -> `由 GitNexus 生成`
- `Loading...` -> `加载中...`
- `Page not found` -> `页面不存在`

### 8. CLI help 和 README

更新 CLI i18n 文案，使 `--lang` 示例包含：

```text
english, chinese, zh-CN, 中文, spanish, japanese
```

同时更新：

- 根 README
- `gitnexus/README.md`

新增中文 wiki 使用示例。

## 环境问题和处理

本地环境是 Termux/Android。

初次运行测试失败，因为项目依赖未安装，`vitest` 不存在。

随后执行：

```bash
npm install --force --ignore-scripts
```

原因是 `onnxruntime-node` 不支持 Android 平台，普通 `npm install` 会被 `EBADPLATFORM` 拦截。

安装后，既有 wiki grouping 测试仍无法完整运行，原因包括：

- `@ladybugdb/core` native binding 在当前环境缺失。
- `gitnexus-shared` 初始未构建。

为避免让新增测试依赖 Android 不可用的 native binding，新增的 `wiki-localization.test.ts` 对 graph 和 LLM 依赖做了 mock。

后续构建时：

```bash
npm run build
```

已经通过：

- `gitnexus-shared` 编译
- `gitnexus` TypeScript 编译

完整 build 后续失败在 `gitnexus-web` 的 `npm ci`：

```text
EALLOWSCRIPTS
--allow-scripts is not allowed in project-scoped installs
```

这是当前 npm/Termux 配置问题，不是本次 wiki 修改的类型错误。

## 当前使用方式

生成中文 wiki：

```bash
gitnexus wiki --lang zh-CN --force
```

或者：

```bash
gitnexus wiki --lang 中文 --force
gitnexus wiki --lang chinese --force
```

默认英文行为：

```bash
gitnexus wiki
```

默认行为保持不变。
